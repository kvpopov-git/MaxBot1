# Правила проекта MaxBot1

Документ задаёт соглашения для разработки и сопровождения чат-бота под мессенджер **MAX** (Node.js).

## Назначение

- Бот работает через официальную библиотеку **`@maxhub/max-bot-api`**.
- Поддерживается **отложенный постинг в канал** на стороне приложения (в API MAX нет встроенной отложенной отправки).

## Стек и запуск

- **Node.js** (ESM): в `package.json` указано `"type": "module"`.
- Точка входа: **`index.js`**. Запуск: `npm start` или `node index.js`.
- Конфигурация среды: **`dotenv`**, файл **`.env`** в корне (не коммитить).

## Переменные окружения

| Переменная         | Обязательность | Описание |
|--------------------|----------------|----------|
| `BOT_TOKEN`        | Да             | Токен бота с [dev.max.ru](https://dev.max.ru) / business.max.ru. |
| `CHANNEL_ID`       | Для постинга   | Числовой `chat_id` канала публикации (в т.ч. отрицательный, если так отдаёт MAX); бот — администратор с правом публикации. |
| `STORAGE_CHAT_ID`  | Да             | `chat_id` чата/канала-хранилища: там хранится очередь отложенных постов и статусы публикации. |
| `ADMIN_USER_IDS`   | Для постинга   | Список `user_id` через запятую; только они используют команды `/post ...`. |
| _(нет)_            |                | Версия и номер сборки хранятся в **`version.json`** и автоматически обновляются CI (GitHub Actions). |

Образец: **`.env.example`** (без секретов).

## Git и GitHub

- Проект подключён к **Git**: ветка по умолчанию **`main`**, первый коммит содержит исходники без `node_modules`, `.env`, `data/`.
- Учётная запись GitHub для публикации: **kvpopov-git**; готовые URL для `remote` — в **`README.md`** (репозиторий примерно `MaxBot1`, см. README если имя на GitHub другое).
- Создание репозитория на GitHub, `remote` и `push` описаны в **`README.md`** (раздел «Git и GitHub»).
- Git для этого ПК установлен на диск **E:** (`E:\Git`; в системном `PATH`: `E:\Git\cmd`). После смены `PATH` **перезапустите терминал или Cursor**; при сбое вызова можно указать полный путь: `E:\Git\cmd\git.exe`.
- Задайте **реальные** `user.name` и `user.email` для коммитов (`git config`), если заменили временные значения в этом клоне.

## Структура репозитория

- **`index.js`** — регистрация команд и обработчиков, запуск бота и планировщика.
- **`scheduler.js`** — очередь отложенных постов в канал, тикер, хранение постов в **`STORAGE_CHAT_ID`** (сам мессенджер MAX).
- **`postDraft.js`** — состояния черновика поста (`/post on` → контент → `/post time`), TTL и парсинг времени.
- **`imageStore.js`** — чтение вложений поста (изображения/видео), обработка **sharp** (JPEG, ресайз для изображений), временное сохранение в **`data/uploads/`**.
- **`admin.js`** — проверка прав по `ADMIN_USER_IDS`.
- **`data/`** — временные локальные файлы изображений (**`uploads/`**) до загрузки в MAX; каталог в **`.gitignore`**.
- **`version.json`** — текущие `version` и `build`; файл **автоматически обновляется** CI (GitHub Actions: `.github/workflows/ci.yml`). При каждом пуше в `main`/`master` `build` увеличивается на 1 и коммитится обратно.
- **`.github/workflows/ci.yml`** — GitHub Actions workflow: установка зависимостей, проверка синтаксиса, инкремент `build` в `version.json`, коммит и пуш.

## Команды бота (контракт)

- `/start` — краткий список команд.
- `/help` — пошаговая настройка: `.env`, `CHANNEL_ID` (канал), `ADMIN_USER_IDS` (кто планирует посты).
- `/version` — показать текущую версию и номер сборки.
- `/my_id` — показать `user_id` (для настройки `ADMIN_USER_IDS`).
- `/time` — показать текущее локальное время сервера и его часовой пояс.
- `/chat_id` — показать `chat_id` и тип текущего чата/канала из контекста сообщения (для настройки `CHANNEL_ID`).
- `/post on` — включить режим формирования черновика.
- Следующим сообщением отправляется содержимое поста: текст и/или вложения (изображения/видео).
- `/post time <спецификация>` — завершить формирование и запланировать:
  - точное время: `yyyy-mm-dd HHhMMm`,
  - относительное: `+1h 30m`, `+90m`, `+2h`.
- После `/post time` бот возвращает идентификатор поста (`id`).
- `/post list [asc|desc]` — список запланированных постов (id, время, первая строка сообщения), сортировка по времени.
- `/post delete <id>` — удалить пост по идентификатору.
- `/post new time <id> <спецификация>` — изменить время выбранного поста:
  - точное: `yyyy-mm-dd HHhMMm`,
  - относительное: `+1h 30m` (смещение считается от старого времени поста).
- `/post move <id> <спецификация>` — алиас `/post new time ...` с теми же правилами.
- `/post off` — сбросить текущий черновик.

Новые команды документируйте здесь и в тексте `/start`, если они публичные.

## Правила кода

- Писать на **современном JavaScript** в стиле текущих файлов: `import`/`export`, `async`/`await`.
- Не подключать лишние зависимости без необходимости; перед добавлением пакета — кратко зафиксировать причину в коммите или PR.
- Ошибки API и планировщика логировать в консоль (`console.error`); не логировать токены и полные `.env`.
- Секреты и `data/` не коммитить; для примеров использовать только **`.env.example`**.

## Планировщик

- Интервал проверки очереди зафиксирован в **`scheduler.js`** (`TICK_MS`); менять осознанно (баланс точности и нагрузки).
- Единственное хранилище отложенных постов — чат/канал **`STORAGE_CHAT_ID`** в MAX. Посты хранятся там с `id` и временем публикации; после успешной публикации удаляются из чата-хранилища.
- Локальные файлы (`data/uploads/`) используются только как временный буфер для обработки изображений перед загрузкой в MAX; видео хранится по токенам MAX в архивном сообщении.
- Зависимость **`sharp`** нужна для обработки изображений; на Linux-хостинге должны ставиться нативные биндинги (типичный образ Node, не «FROM scratch» без libc).

## Правила для Cursor (`.cursor/rules`)

- В каталоге **`.cursor/rules/`** лежат файлы **`.mdc`** с подсказками для ИИ в Cursor (Project Rules в документации Cursor).
- Они дополняют этот документ: краткий контекст по стеку, отложенному постингу и секретам, чтобы ассистент не предлагал устаревшие или небезопасные решения.
- **Когда обновлять `.cursor/rules`:** при смене архитектуры (новые модули, другой способ планирования), новых обязательных env, смене библиотеки MAX или контракта команд — правки вносите **параллельно** с кодом и с **`PROJECT_RULES.md`**, иначе подсказки в редакторе разойдутся с реальным проектом.

## Изменения правил (общая дисциплина)

| Что меняется | Что обновить |
|--------------|----------------|
| Поведение бота, команды, env | `index.js`, **`PROJECT_RULES.md`**, текст `/start`, при необходимости **`.cursor/rules/*.mdc`** |
| Структура каталогов, новые файлы | **`PROJECT_RULES.md`**, при смысле для ИИ — **`.cursor/rules`** |
| Только стиль кода под конкретные файлы | можно добавить отдельный `.mdc` с `globs` (например `**/*.js`) |

**`PROJECT_RULES.md`** — основной человекочитаемый свод правил репозитория. **`.cursor/rules`** — сжатый контекст для Cursor; его нужно **поддерживать в актуальном состоянии** вместе с основным файлом, а не вместо него.
